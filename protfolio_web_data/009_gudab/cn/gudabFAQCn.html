gudab FAQ

gudab 的config目录下，每个档案的用途?

  1. ./id/gudab.id：给机器看的gudab ID
  2. ./id/im_ .worker：给人看的gudab ID (档名的XXX)
  3. ./metrics_os_fetcher/linux_Fetcher_ ..yml：取得 Linux 指标的 Commad和 Regular Express的设定。
  4. ./metrics_os_fetcher/linux_SSH_default_Info.properties：gudab SSH连线到被监控对象主机的 Linux Service Account 设定。
  5. ./gudab.properties：gudab 内建的 consoleDB 和Mail Server设定。
  6. ./log4j2.xml：gudab 日志设定。

gudab 突然无法重启 (没有画面)?

请优先与我们联络。
若您想尝试自行处理，下列2种可能原因与解决方法：

  1. ID问题：将gudab/config/id 目录内档案清除，重新启动。
  2. ConsoleDB问题：尝试将 gudab/consoleDB/data 目录内档案清除，重新启动。注意：此步骤gudab会初始化，遗失所有指标、备份、设定等。

Linux Service Accounts、MongoDB Service Account 及 gudab Service Account，预设或建议的帐密为? 浏览器预设登入的帐密，是哪一种?

  1. Linux Service Accounts：(gudab/gudab) 建议
  2. MongoDB Service Accounts：(root/gudab) 建议
  3. gudab Service Accounts：(root/root) 预设
  4. 浏览器预设登入的帐密是 gudab Service Account。

为什么要在被监控对象的主机上，建立 Linux Service Account?

gudab透过SSH连结被监控对象，考量被监控对象所在的主机(Linux Host)的权限、安全、稽核等因素，建议所有主机统一新增专属Linux Service Account (eg. gudab)。

忘记登入密码，执行首页的忘记密码，却无法解决 ?

请优先与我们联络。
若您想尝试自行处理，下列2种可能原因与解决方法：

  1. 设定问题：收不到信。不论内网或外网，gudab要能连结到某台Mail Server，并在gudab.properties输入寄件人的资讯。
  2. 信箱问题：帐号与email有误。此处 email是指注册时填写的信箱。
    预设的root帐号，其初始Email状态为空，请尽早更新使用者资料。

gudab Account分成几种角色(权限)?

  1. root user：是预设用户、名称唯一、不可删除、只能加admin用户。第一次登入，请记得更新使用者资料 (填写email、修改密码)。
  2. admin group：全部功能。
  3. power user group：除了加用户，全部操作功能。
  4. operator group：看系统监控，无法新增伺服器；可开关告警 (发alert)，无法修改告警设定。

多用户，是否可以使用不同的 DB 帐密，监控同一对象。

不同 gudab 用户，均可以新增被监控对象。但 gudab 为了避免多用户监控同一对象，造成重复监控的情况，故采用第一次联机的 DB 帐密建立设定 (后面的用户无法改动)。
范例如下：

  1. gudab 用户 1(root)，先使用 DB 帐密(root/12345)，加入监控 Replica SetA。
  2. gudab 用户 2(test)，后使用 DB 帐密(test/67890)，加入监控 Replica SetA。
  3. gudab 用户 2 输入的 DB 帐密正确(test/67890)，但实际是拿用户 1 的帐密 (root/12345)去做监控。

注：第一次联机的 DB 帐密，应该采用专属的 MongoDB Service Account。

为什么要在被监控对象上，建立MongoDB Service Account ?

  1. 为避免监控 n 个成员的丛集，需输入 n 次成员的信息，故 gudab 提供的 Auto Discovery 功能，能透过一组帐密连结所有丛集成员(需注意这组 DB 帐密是否有足够的管理权限)。
  2. 另考虑被监控对象的权限、安全、稽核等，建议所有被监控对象统一新增专属的 MongoDB Service Account (eg. root)。

注：丛集是指 Resplica Set (RS)、Sharded Cluster (SC)。

Auto Discovery 无法加入全部的丛集成员?

请优先与我们联络。
若您想尝试自行处理，下列4种可能原因与解决方法：

  1. 确认是否建立gudab用的MongoDB Service Account。
  2. RS必须从 Primary或Secondary做Auto Discovery，不可以从 Arbiter。
  3. SC必须从 Router(mongos)做Auto Discovery，不可以从分片成员。
  4. gudab 加入监控时，会取得 hostname，所以 gudab 主机需与被监控对象有相同 DNS 或在 hosts 档案加入对应 IP。

为什么 gudab没有看到 Host Status 监控项目 (CPU, Memory, Storage, Swap)?

请优先与我们联络。
若您想尝试自行处理，下列2种可能原因与解决方法：

  1. 此监控项目，若被监控对象安装在 Windows 上，则 gudab 暂不支持。
  2. 尚未设定 linux_SSH_default_Info.properties 的信息。

除了现有的监控指标，能否加入其他项目指标?

可以，这些属于客制化服务范畴。

除了发信告警，是否提供其他告警方式?

可以，这些属于客制化服务范畴。

gudab 的备份方式?

  1. 对象：Replica Set 成员
  2. 完整备份(全备份、快照)：建立 delayed 成员做全备份，预设延迟3600 秒。(需手动复制 data files 重启环境、尚不支持快照)
  3. 增量备份：持续备份 Replica Set 成员的 oplog。

gudab采用哪种备份策略? (哪个DB成员作为备份对象)

gudab 采用 Secondary Prefer策略。
优先从任一个 Secondary成员作为对象备份oplog，若无法连线Secondary，才会改从Primary备份。

MongoDB Standalone (Single node) 能否做增量备份 ?

Standalone要做增量备份，需初始成单一成员的RS，作为Primary产生oplog，gudab便能做增量备份。

gudab备份的注意事项?

  1. 第一次启动备份，无法备份之前的oplog，只会备份启动时间之后的oplog。之后会产生 tail collection记录最后一笔备份的时间，之后关闭备份再启动，会接续备份记录最后一笔时间之后的oplog(即使是在gudab关闭备份时产生的)。
  2. 在备份 Sharded Cluster 时：
    1. 不支持 Config Server 异动。
      因此，备份前须做好分片的 Zone Range，并在 mongos 执行指令：
      1. sh.disableAutoSplit()
      2. sh.setBalancerState(false)
    2. 备份前，需先做完整备份。
    3. 分片须为 Replica Set。

gudab 还原的注意事项?

  1. 被监控对象，必须先透过完整备份的档案重新启动，才能执行gudab还原。
  2. 目前还原起讫的设定值，为每小时的整点钟和半点钟。

启动备份后，新增的 Collection无法备份加入之前的 oplog?

若当初建立备份设定，是指定要备份的 Collection (非勾选所有资料库)，若有新的colleciton要加入原备份设定，则须修改后重新执行新增备份设定。

gudab 备份多久的 oplog ? 是否有保护机制，避免硬碟空间不足?

  1. gudab预设备份 7日的 oplog。
  2. 用户须自行评估 oplog的成长量，选择适当容量的储存空间；或修改 config file的备份天数 (调整后须重启 Gudab)。

可否指定要备份/还原的 Collection?

  1. 备份：目前可指定 Collection。
  2. 还原：目前暂不支持指定 Collection，会一次还原全部。

MongoDB 已有Replica Set记录oplog，为什么还需要gudab 备份oplog?

可以从2个面向来看：

MongoDB's Replica Set gudab
备份上限 capped collection的大小 硬碟空间和备份天数
还原方式 Mongo Shell (输入指令) 操作介面 (滑鼠点击)
  1. 备份上限：假设资料量为1TB，记录oplog 的capped colletion为50GB，若是人为失误操作eg. db.coll.remove({ })，会产生大量oplog把旧有的oplog覆盖。即使发生人为失误，只要不超过硬碟上限，就可以继续保留，直到超过备份天数。而且主要由gudab备份oplog，MongoDB的capped collection就不用设定太大，保留运行环境的硬体资源。
  2. 还原方式：gudab提供操作介面，就不需要查找相对应的指令和步骤。
  3. 另外，gudab其独立于MongoDB架构之外，可安装在不同的系统环境OS、机房位置，采用不同的系统帐密、目录等管理权限，能提供更严谨的安全性，在储存备份资料上，等于多一份保障。

  2. gudab Enterprise 取决于购买订阅的 data nodes 数量。

如果要达到分散容错的丛集架构，是否有最小主机数量?

以 Replica Set 为例，建议您最小主机数量为 3。
架构：Svr1(Primary) + Svr2(Secondary) + Svr3(Arbiter & gudab)
